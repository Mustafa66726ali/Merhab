# دليل اختبار الأدوار والصلاحيات — نظام مرحّاب

> **الغرض:** ملف اختبار يدوي (UAT) للتحقق من أن كل دور يعمل كما يجب.  
> **لكل دور ثلاثة أقسام:** نبذة → صلاحيات (نقاط اختبار) → تقرير متوقع بعد الاختبار.

---

## كيفية استخدام هذا الملف

1. سجّل دخول بحساب الدور المطلوب.
2. نفّذ كل نقطة اختبار بالترتيب.
3. ضع ✓ أو ✗ بجانب كل نقطة.
4. قارِن النتيجة بقسم **«التقرير المتوقع»** في نهاية كل دور.

**حسابات مرجعية للتجربة:**

| الدور | مسار الدخول |
|-------|-------------|
| مدير النظام | `/login` → `/dashboard` |
| مدير المنصة | `/login` → `/platform/dashboard` |
| مدير الفعالية | `/login` → `/event-manager/dashboard` |
| منظم الفعالية | `/login` → `/event-organizer/dashboard` |
| المنسّق | `/login` → `/coordinator/check-in` |
| مدير الدخول | `/login` → `/entry-manager/check-in` |
| الضيف | رابط `/i/{token}` — بدون تسجيل دخول |

# 1. مدير النظام (`system_manager`)

## 1.1 نبذة عن الدور

مدير النظام (السوبر أدمن) هو المسؤول الأعلى عن المنصة ككل. يدير المنصات المتعددة، المحتوى العام للموقع، التكاملات، والإعدادات على مستوى النظام. لا يقتصر على منصة واحدة — يرى كل شيء.

## 1.2 نقاط الاختبار — الصلاحيات

### إدارة المنصات
- [ ] عرض قائمة كل المنصات (`/platforms`)
- [ ] فتح صفحة تفاصيل منصة (`/platforms/{id}`)
- [ ] **إضافة** منصة جديدة (`/platforms/add`) — اسم، مالك، بريد، كلمة مرور
- [ ] التحقق من إنشاء حساب **مدير منصة** تلقائياً عند الإضافة
- [ ] **تعديل** بيانات منصة
- [ ] **حظر/تفعيل** منصة
- [ ] إرسال **رسالة** لمالك المنصة من صفحة المنصة
- [ ] إرسال **إشعار** لمالك المنصة

### لوحة التحكم والتقارير
- [ ] عرض لوحة التحكم الرئيسية (`/dashboard`)
- [ ] عرض KPIs وإحصائيات النظام
- [ ] فتح التقارير (`/reports`)
- [ ] تصدير تقارير (إن وُجد)

### المحتوى العام للموقع
- [ ] إدارة **صفحة الهبوط** (`/landing-page`)
- [ ] إدارة **الصفحات الثابتة** (`/static-pages`)
- [ ] إدارة **الأسئلة الشائعة** (`/faq`)
- [ ] إدارة **الإعلانات والبانرات** (`/announcements`)
- [ ] إدارة **الوسائط العامة** (`/public-media`)
- [ ] إدارة **الروابط الخارجية** (`/external-links`)

### النظام والتكاملات
- [ ] فتح **التكاملات** (`/integrations`) — واتساب، Twilio، إلخ
- [ ] فتح **الإعدادات** (`/settings`)
- [ ] فتح **النسخ الاحتياطي** (`/backup`)
- [ ] عرض **سجلات النشاط** (`/activity-logs`)
- [ ] عرض **أداء الخوادم** (`/servers-performance`)

### المراسلة
- [ ] فتح صندوق الرسائل (`/messages`)
- [ ] **إرسال رسالة** إلى أي مدير منصة
- [ ] التأكد أنه **لا يمكن** مراسلة مدير فعالية مباشرة (إن طُبّق القيد)
- [ ] عرض الإشعارات (`/notifications`)

### ما لا يجب أن يفعله (اختبار سلبي)
- [ ] محاولة الدخول بحساب ضيف → يُرفض من `/login`

## 1.3 التقرير المتوقع بعد الاختبار

بعد اكتمال الاختبار، يجب أن يكون مدير النظام قادراً على:

- **إنشاء منصة كاملة** مع حساب مدير منصة جاهز للدخول.
- **مراقبة** كل المنصات والفعاليات من مستوى النظام.
- **إدارة المحتوى العام** للموقع (هبوط، FAQ، إعلانات) دون الاعتماد على أحد.
- **مراسلة** أي مدير منصة واستلام ردوده.
- **الوصول** لكل صفحات الإعدادات والتكاملات والنسخ الاحتياطي.
- **عدم التقيّد** بمنصة واحدة — يرى بيانات النظام بالكامل.

**معايير النجاح:** جميع نقاط «إضافة / عرض / تشغيل» أعلاه تعمل دون خطأ 403 أو 404.

# 2. مدير المنصة (`platform_admin`)

## 2.1 نبذة عن الدور

مدير المنصة هو مالك منصة واحدة (شركة/جهة). يدير فعاليات منصته، أعضاءها، التقارير، والإعدادات. يُنشأ حسابه تلقائياً عند إضافة المنصة من قبل مدير النظام.

## 2.2 نقاط الاختبار — الصلاحيات

### لوحة المنصة
- [ ] عرض لوحة تحكم المنصة (`/platform/dashboard`)
- [ ] عرض KPIs: فعاليات، ضيوف، نسب تأكيد، RSVP
- [ ] عرض قائمة الفعاليات الأخيرة

### إدارة الفعاليات
- [ ] عرض كل فعاليات المنصة (`/platform/events`)
- [ ] **إضافة** فعالية جديدة
- [ ] **تعديل** فعالية (عنوان، تاريخ، موقع، صورة)
- [ ] فتح تفاصيل فعالية (`/platform/events/{id}`)
- [ ] **تشغيل** فعالية (بدء) و**إيقاف**ها (إنهاء) — إن وُجد الزر

### الأعضاء والطاقم
- [ ] عرض الأعضاء (`/platform/users`)
- [ ] **إضافة** مدير فعالية (`event_manager`)
- [ ] **إضافة** منظم فعالية (`event_organizer`)
- [ ] **تعديل** صلاحيات العضو: مسح QR / تعديل ضيوف / إرسال رسائل
- [ ] **تعطيل** أو إزالة عضو

### الضيوف والتقارير
- [ ] عرض ضيوف المنصة (`/platform/guests`)
- [ ] فتح التقارير (`/platform/reports`)
- [ ] تصدير PDF لتقرير فعالية (إن وُجد)

### الإعدادات
- [ ] فتح إعدادات المنصة (`/platform/settings`)
- [ ] **تعديل** شعار المنصة
- [ ] **تعديل** رقم واتساب المنصة
- [ ] **تفعيل/إيقاف** دعوات واتساب

### المراسلة والإشعارات
- [ ] عرض الرسائل (`/platform/messages`)
- [ ] **إرسال رسالة** إلى مدير النظام
- [ ] عرض الإشعارات (`/platform/notifications`)
- [ ] التأكد أنه **لا يستطيع** مراسلة مدير منصة آخر

### ما لا يجب أن يفعله (اختبار سلبي)
- [ ] محاولة فتح `/platforms` (إدارة كل المنصات) → مرفوض أو غير ظاهر
- [ ] محاولة فتح `/settings` (إعدادات النظام) → مرفوض
- [ ] محاولة رؤية فعاليات منصة أخرى → مرفوض

## 2.3 التقرير المتوقع بعد الاختبار

بعد الاختبار، يجب أن يكون مدير المنصة قادراً على:

- **إدارة منصته بالكامل** دون الوصول لمنصات أخرى.
- **إنشاء فعاليات** وتعيين مديري ومنظمي فعاليات عليها.
- **ضبط صلاحيات** كل عضو (QR، ضيوف، رسائل) من واجهة الأعضاء.
- **متابعة الأداء** عبر التقارير ولوحة التحكم.
- **التواصل** مع مدير النظام فقط عبر الرسائل.
- **تخصيص** هوية المنصة (شعار، واتساب).

**معايير النجاح:** كل العمليات تخص منصته فقط؛ لا تسريب بيانات منصات أخرى.

# 3. مدير الفعالية (`event_manager`)

## 3.1 نبذة عن الدور

مدير الفعالية هو المسؤول الكامل عن فعالية أو عدة فعاليات داخل المنصة. يبني هيكل الفعالية (أقسام، مجموعات، جدول)، يدير الضيوف والدعوات والجلوس، ويُنشئ حسابات المنسّق ومدير الدخول.

## 3.2 نقاط الاختبار — الصلاحيات

### لوحة الفعالية
- [ ] عرض لوحة التحكم (`/event-manager/dashboard`)
- [ ] عرض فعالياته المعيّنة
- [ ] فتح تفاصيل فعالية (`/event-manager/events/{id}`)

### هيكل الفعالية (حصري على مدير الفعالية)
- [ ] **إضافة** قسم (`/events/{id}/groups` أو الأقسام)
- [ ] **تعديل** قسم / **حذف** قسم
- [ ] **إضافة** مجموعة وربطها بقسم
- [ ] **إضافة** بند في الجدول الزمني (`/events/{id}/schedule`)
- [ ] **تعديل** و**حذف** بنود الجدول

### الضيوف
- [ ] عرض ضيوف الفعالية (`/event-manager/events/{id}/guests`)
- [ ] **إضافة** ضيف يدوياً
- [ ] **تعديل** بيانات ضيف
- [ ] **حذف** ضيف
- [ ] **استيراد** ضيوف (Excel/CSV إن وُجد)
- [ ] عرض حالات الضيوف: مدعو / مؤكد / حضر / جلس / معتذر

### الدعوات
- [ ] فتح صفحة الدعوات (`/event-manager/events/{id}/invitations`)
- [ ] **تحرير** نص الدعوة
- [ ] **إرسال** دعوة لضيف / مجموعة / الكل
- [ ] **إرسال تذكير** للضيوف
- [ ] التحقق من وصول رابط فريد `/i/{token}` لكل ضيف

### توزيع الجلوس
- [ ] فتح توزيع المقاعد (`/event-manager/events/{id}/seating`)
- [ ] **إنشاء** مخطط جلوس
- [ ] **إضافة** طاولة ومقاعد
- [ ] **إسناد** ضيف لمقعد يدوياً
- [ ] **إزالة** ضيف من مقعد

### البث المباشر
- [ ] فتح صفحة البث (`/event-manager/events/{id}/broadcast`)
- [ ] **تفعيل** البث للضيوف
- [ ] **اختيار** نوع البث: يوتيوب / ملف صوتي / ميكروفون / كاميرا
- [ ] **حفظ** إعدادات البث
- [ ] **بدء** بث مباشر (ميكروفون/كاميرا)
- [ ] **إرسال** رابط البث للحاضرين والجالسين عبر واتساب
- [ ] **نسخ** رابط البث العام `/live/{token}`

### تشغيل الفعالية
- [ ] **بدء** الفعالية (تصبح «تعمل الآن»)
- [ ] **إنهاء** الفعالية (تصبح «منتهية»)

### الطاقم التشغيلي
- [ ] فتح «المنسقون ومدراء الدخول» (`/event-manager/staff`)
- [ ] **إضافة** حساب منسّق (رجال/نساء)
- [ ] **إضافة** حساب مدير دخول
- [ ] **تعيين** موظف على فعالية محددة
- [ ] **إلغاء** تعيين موظف من فعالية

### فريق العمل والرسائل
- [ ] عرض فريق العمل (`/event-manager/team`)
- [ ] فتح الرسائل (`/event-manager/messages`)
- [ ] **إرسال** رسالة لضيف
- [ ] فتح تبويب «تهنئات واستفسارات» ضمن الرسائل
- [ ] **عرض** قائمة تهنئات الضيوف (اسم الضيف + التاريخ + نص التهنئة)
- [ ] **تصفية** التهنئات حسب الفعالية والتاريخ والبحث بالاسم
- [ ] **طباعة** تقرير التهنئات كـ PDF من نفس الصفحة
- [ ] **تصدير** تقرير الفعالية الشامل (PDF) ويظهر فيه قسم التهنئات مزخرفاً
- [ ] عرض استفسارات الضيوف للمنسّق في نفس التبويب

### الإشعارات
- [ ] فتح صفحة الإشعارات (`/event-manager/notifications`)
- [ ] **استلام إشعار** فور إرسال ضيف تهنئة جديدة (جرس الهيدر + العداد)
- [ ] فتح الإشعار وتحديده كمقروء
- [ ] **تحديد الكل كمقروء** وحذف الإشعارات المقروءة

### تقدم الفعالية
- [ ] فتح صفحة التقدم (`/event-manager/events/{id}/progress`)
- [ ] التحقق من نسبة اكتمال التحضيرات

### ما لا يجب أن يفعله (اختبار سلبي)
- [ ] محاولة إدارة فعالية ليست ضمن صلاحياته → 403
- [ ] محاولة إضافة منصة جديدة → مرفوض

## 3.3 التقرير المتوقع بعد الاختبار

بعد الاختبار، يجب أن يكون مدير الفعالية قادراً على:

- **بناء فعالية كاملة** من الصفر: أقسام، مجموعات، جدول، ضيوف، دعوات، جلوس.
- **إرسال دعوات** واستلام تأكيدات الضيوف عبر الروابط العامة.
- **تشغيل وإيقاف** الفعالية في الوقت المناسب.
- **إدارة البث** وإيصاله للضيوف الحاضرين.
- **توفير فريق تشغيل** (منسّق + مدير دخول) معيّن على فعالياته.
- **متابعة التقدم** والإحصائيات لحظياً.
- **عرض وطباعة** تهنئات الضيوف من حساب مدير الفعالية، وإدراجها في تقرير المناسبة PDF.
- **استلام إشعارات** فورية عند وصول تهنئة جديدة من الضيوف.

**معايير النجاح:** دوره هو «المدير التنفيذي» للفعالية — كل ما عدا إدارة المنصة والنظام.

# 4. منظم الفعالية (`event_organizer`)

## 4.1 نبذة عن الدور

منظم الفعالية يعمل على تنفيذ الفعالية يومياً: ضيوف، دعوات، حضور، رسائل، وبث. لا يبني هيكل الفعالية (أقسام/جدول) — ذلك من صلاحية مدير الفعالية. صلاحياته الإضافية (QR، تعديل ضيوف، رسائل) تُضبط من مدير المنصة.

## 4.2 نقاط الاختبار — الصلاحيات

### لوحة التحكم
- [ ] عرض لوحة المنظم (`/event-organizer/dashboard`)
- [ ] عرض الفعاليات المعيّنة عليه فقط

### الضيوف (يتطلب `perm_edit_guests`)
- [ ] عرض الضيوف (`/event-organizer/guests` أو ضمن فعالية)
- [ ] **إضافة** ضيف — إن مُنحت الصلاحية
- [ ] **تعديل** ضيف
- [ ] محاولة الإضافة **بدون** الصلاحية → 403

### الدعوات (يتطلب `perm_send_messages`)
- [ ] فتح الدعوات (`/event-organizer/invitations`)
- [ ] **تحرير** نص دعوة
- [ ] **إرسال** دعوة / تذكير
- [ ] محاولة الإرسال **بدون** الصلاحية → الزر مخفي أو 403

### تسجيل الحضور (يتطلب `perm_scan_qr`)
- [ ] فتح تسجيل الحضور (`/event-organizer/check-in`)
- [ ] **مسح** QR ضيف → يصبح «حضر»
- [ ] **البحث** عن ضيف بالاسم وتسجيل حضوره
- [ ] محاولة المسح **بدون** الصلاحية → الصفحة غير متاحة

### البث المباشر
- [ ] فتح البث (`/event-organizer/broadcast`)
- [ ] **تفعيل** بث يوتيوب أو ملف صوتي
- [ ] **إرسال** رابط البث للحاضرين

### الجلوس
- [ ] فتح توزيع المقاعد (`/event-organizer/seating`)
- [ ] **عرض** المخططات والطاولات
- [ ] **إسناد** ضيف لمقعد (إن سُمح له)

### الرسائل وكلمات الشكر
- [ ] فتح الرسائل (`/event-organizer/messages`) — إن مُنحت الصلاحية
- [ ] **إرسال** رسالة لضيف
- [ ] عرض كلمات الشكر (`/event-organizer/gratitudes`)
- [ ] عرض **تهنئات** و**استفسارات** الضيوف من الدعوة

### الجدول والأقسام (اختبار سلبي — يجب أن يُرفض)
- [ ] محاولة **إضافة** قسم جديد → 403
- [ ] محاولة **إضافة** بند جدول زمني → 403
- [ ] محاولة **بدء/إنهاء** الفعالية → 403
- [ ] محاولة **إضافة** منسّق من صفحة الطاقم → 403

### الشريط الجانبي — ظهور الروابط حسب الصلاحية
- [ ] رابط «تسجيل الحضور» يظهر فقط مع `perm_scan_qr`
- [ ] رابط «الدعوات» يظهر فقط مع `perm_send_messages`
- [ ] رابط «البث المباشر» يظهر دائماً

## 4.3 التقرير المتوقع بعد الاختبار

بعد الاختبار، يجب أن يكون منظم الفعالية قادراً على:

- **تشغيل الفعالية يوم الحفل**: دعوات، حضور، بث، رسائل.
- **تنفيذ ما يُسمح له** حسب الصلاحيات الثلاث الممنوحة له.
- **عدم تعديل** هيكل الفعالية (أقسام، مجموعات، جدول) — وهذا سلوك صحيح.
- **مشاهدة** تقدم الفعالية وضيوفها ضمن نطاق فعالياته فقط.

**معايير النجاح:** الواجهة تعكس صلاحياته بدقة (إظهار/إخفاء الروابط)، ولا يتجاوز حدود دوره.

# 5. المنسّق (`staff` + `coordinator`)

## 5.1 نبذة عن الدور

المنسّق (رجال أو نساء) يعمل داخل القاعة. مهمته الأساسية: **إجلاس الضيوف** في مقاعدهم بعد تسجيل حضورهم. يمسح QR الضيف عند الطاولة أو يُدخله يدوياً في المقعد. يُعيَّن على فعاليات محددة فقط.

## 5.2 نقاط الاختبار — الصلاحيات

### الدخول والنطاق
- [ ] تسجيل الدخول → التوجيه إلى `/coordinator/check-in`
- [ ] عرض شارة الدور: «منسّق» (رجال/نساء إن وُجد التصنيف)
- [ ] رؤية **فعالياته المعيّنة فقط** — وليس كل فعاليات المنصة

### تسجيل الحضور (إن مُنح `perm_scan_qr`)
- [ ] فتح تسجيل الحضور (`/coordinator/check-in`)
- [ ] **مسح** QR ضيف عند البوابة → «حضر»
- [ ] التأكد أن الضيف **لا يُجلَس** تلقائياً عند البوابة — فقط «حضر»

### إجلاس الضيوف (الصلاحية الأساسية)
- [ ] فتح صفحة الإجلاس (`/coordinator/seating`)
- [ ] **عرض** الطاولات والمقاعد الفارغة والمشغولة
- [ ] **إدراج يدوي**: اختيار ضيف → إسناده لمقعد → حالته «جلس»
- [ ] **مسح QR** عند الطاولة (`scan-seat`):
  - [ ] ضيف «حضر» → يُجلَس بنجاح → «جلس»
  - [ ] ضيف «مؤكد» فقط (لم يحضر بعد) → **رفض** مع رسالة «سجّل حضوره أولاً»
- [ ] **إزالة** ضيف من مقعد

### الإشعارات
- [ ] **استلام إشعار** عند إرسال ضيف استفساراً من صفحة الدعوة
- [ ] فتح الإشعار وقراءة نص الاستفسار واسم الضيف

### ما لا يجب أن يفعله (اختبار سلبي)
- [ ] محاولة **إضافة** ضيف جديد → 403 (بدون `perm_edit_guests`)
- [ ] محاولة **إرسال** دعوة → 403 (بدون `perm_send_messages`)
- [ ] محاولة فتح فعالية **غير معيّنة** عليه → لا بيانات / 403
- [ ] محاولة **إنشاء** مخطط جلوس جديد → مرفوض (حسب التصميم)

## 5.3 التقرير المتوقع بعد الاختبار

بعد الاختبار، يجب أن يكون المنسّق قادراً على:

- **إجلاس الضيوف** بسرعة عبر المسح أو الإدراج اليدوي.
- **التمييز** بين «حضر» (عند البوابة) و«جلس» (عند الطاولة).
- **رفض** إجلاس ضيف لم يُسجَّل حضوره بعد — برسالة واضحة.
- **العمل فقط** على فعالياته المعيّنة.

**معايير النجاح:** تدفق الحضور → الإجلاس يعمل بسلاسة؛ لا وصول لإدارة الضيوف أو الدعوات إلا إن مُنحت صلاحيات إضافية.

# 6. مدير الدخول (`staff` + `entry_manager`)

## 6.1 نبذة عن الدور

مدير الدخول يقف عند بوابة القاعة. مهمته: **تسجيل وصول الضيف** بمسح QR الدعوة. يحوّل الضيف من «مؤكد» إلى «حضر». لا يُجلِس الضيوف — ذلك من مهمة المنسّق.

## 6.2 نقاط الاختبار — الصلاحيات

### الدخول والنطاق
- [ ] تسجيل الدخول → التوجيه إلى `/entry-manager/check-in`
- [ ] عرض شارة الدور: «مدير دخول»
- [ ] رؤية ضيوف **فعالياته المعيّنة فقط**

### تسجيل الحضور (الصلاحية الأساسية)
- [ ] فتح صفحة تسجيل الدخول (`/entry-manager/check-in`)
- [ ] **مسح** QR ضيف مؤكد → يصبح «حضر»
- [ ] **البحث** بالاسم وتسجيل الحضور يدوياً
- [ ] مسح QR لضيف **حضر مسبقاً** → رسالة «مسجّل مسبقاً» دون خطأ
- [ ] مسح QR لضيف **معتذر** → رفض أو تحذير
- [ ] عرض إحصائيات: إجمالي / حضر / بانتظار

### الكاميرا والأمان
- [ ] فتح ماسح QR بالكاميرا (يتطلب HTTPS في الإنتاج)
- [ ] إدخال رمز يدوياً عند تعذّر الكاميرا

### ما لا يجب أن يفعله (اختبار سلبي)
- [ ] محاولة فتح `/coordinator/seating` → مرفوض أو غير متاح
- [ ] محاولة **إجلاس** ضيف في مقعد → 403
- [ ] محاولة **إضافة/تعديل** ضيف → 403
- [ ] محاولة **إرسال** دعوة → 403
- [ ] محاولة الوصول لفعالية **غير معيّنة** → لا بيانات

## 6.3 التقرير المتوقع بعد الاختبار

بعد الاختبار، يجب أن يكون مدير الدخول قادراً على:

- **تسجيل حضور** كل ضيف مؤكد عند الوصول بسرعة (مسح أو بحث).
- **عدم الخلط** بين «حضر» و«جلس» — لا يُجلِس أحداً.
- **العمل** على فعالياته فقط دون رؤية باقي المنصة.
- **التعامل** مع الحالات المكررة (حضر مسبقاً) بهدوء.

**معايير النجاح:** بوابة الدخول سريعة وموثوقة؛ لا صلاحيات زائدة عن المطلوب.

# 7. الضيف (`guest`) — بدون تسجيل دخول

## 7.1 نبذة عن الدور

الضيف لا يملك حساباً في لوحة التحكم. يتفاعل عبر **رابط دعوة فريد** يصله على واتساب أو بريد. يؤكد حضوره، يحصل على QR، ويشاهد البث إن وُجد.

## 7.2 نقاط الاختبار — الصلاحيات

### رابط الدعوة `/i/{token}`
- [ ] فتح الرابط **بدون** تسجيل دخول
- [ ] **عرض** اسم الضيف وعنوان الفعالية وصورة الغلاف
- [ ] **عرض** التاريخ والوقت
- [ ] **عرض** الموقع الكامل (مكان + عنوان جغرافي) وزر الخريطة
- [ ] **عرض** القسم والمجموعة المخصّصة للضيف
- [ ] **عرض** برنامج الحفل (الجدول الزمني)
- [ ] **عرض** من سيحضر من مجموعته

### RSVP (تأكيد / اعتذار)
- [ ] الضيف «مدعو» → زر **تأكيد الحضور** يعمل
- [ ] بعد التأكيد → الحالة «مؤكد» + **ظهور QR**
- [ ] **تنزيل** QR كصورة PNG
- [ ] **تنزيل** QR كملف PDF
- [ ] زر **اعتذار** يعمل والحالة تصبح «معتذر»
- [ ] تغيير الرد من اعتذار → تأكيد (والعكس)

### التهنئة والاستفسار
- [ ] **كتابة** كلمة تهنئة وإرسالها
- [ ] **كتابة** استفسار للمنسّق وإرساله
- [ ] ظهور التهنئة/الاستفسار في لوحة المنظم/المدير
- [ ] بعد التهنئة → يصل **إشعار** لمدير الفعالية
- [ ] بعد الاستفسار → يصل **إشعار** للمنسّق

### البث المباشر
- [ ] ظهور قسم البث عند تفعيله من المنظم
- [ ] **تشغيل** بث يوتيوب
- [ ] **تشغيل** ملف صوتي
- [ ] فتح رابط البث `/live/{token}` و**تشغيل** المحتوى

### ما لا يجب أن يفعله (اختبار سلبي)
- [ ] محاولة فتح `/login` والدخول كضيف → مرفوض
- [ ] محاولة فتح `/dashboard` → إعادة توجيه لـ login
- [ ] فتح رابط دعوة **غير صالح** → رسالة خطأ واضحة
- [ ] QR **قبل** التأكيد → مقفل مع رسالة «سيظهر بعد التأكيد»

## 7.3 التقرير المتوقع بعد الاختبار

بعد الاختبار، يجب أن يكون الضيف قادراً على:

- **فتح دعوته** من الجوال بسهولة دون إنشاء حساب.
- **تأكيد أو الاعتذار** عن الحضور بوضوح.
- **الحصول على QR** فور التأكيد واستخدامه عند البوابة.
- **معرفة** مكان ووقت الفعالية وبرنامجها.
- **إرسال** تهنئة أو استفسار للمنسّق.
- **مشاهدة البث** من الدعوة أو من رابط `/live/...`.

**معايير النجاح:** تجربة ضيف سلسة على الجوال؛ لا حاجة لدعم تقني.
